# Speedracer

Speedracer is a GitHub Action for speculative execution of matrix jobs. Callers add `speedracer: [a, b]` (or more letters) as their last matrix dimension, then call this action as their first step. Multiple copies of the same job race; the fastest runner wins, duplicates cancel themselves.

The entire action is a single file: `action.yaml` (composite action with inline bash). There is no build step, no compiled language, no dependencies beyond `gh` and `jq`.

## How It Works

1. **CPU gate** — Non-`a` copies on slow runners (matched by `slow-cpu-pattern`) abort via `kill -TERM $PPID`. Copy `a` is the fallback and never CPU-gates.
2. **Check-run posting** — Fast copies POST an `in_progress` check-run to the GitHub Checks API with an `external_id` of the form `spec-{letter}-{run_id}-{check_name}`. Slow copy `a` skips posting to avoid tricking fast siblings into deferring.
3. **Sibling polling** — Each copy polls for higher-lettered siblings (`d > c > b > a`). If a higher-priority sibling is `in_progress` or `success`, the lower copy cancels itself.
4. **Post-step** — Uses `gacts/run-and-post-run` to resolve the check-run after the job finishes. `job.status == success` → POST new completed/success check-run. `job.status == failure` → PATCH existing check-run to completed/failure.

The check-run contract is what makes this work with branch protection — the winning copy's check-run satisfies required status checks.

## Key Files

- `action.yaml` — The entire action: inputs, gate logic, check-run management, post-step hook. All logic is inline bash.
- `runner-pool.tsv` — Auto-generated by the profiler workflow. First column of the first data row = the default `slow-cpu-pattern`. Tab-separated: `cpu_id count pct avg_bench_ms avg_disk_mbs cores mem_mb full_model`. The profiler only commits this file when the CPU pool composition changes (new type appears or unseen slowest type is pruned) — benchmark number fluctuations are updated in-place but don't trigger commits.
- `.github/workflows/runner-profile.yaml` — Weekly cron (Monday 09:37 UTC) that benchmarks 10 runners and merges results into `runner-pool.tsv`. CPU types accumulate over time; unseen types are only pruned from the top (slowest) until a seen entry is reached.
- `.github/workflows/speedracer-test.yaml` — Comprehensive test suite (20+ test jobs) that validates both job outcomes and Checks API state.

## Testing

Tests run on push/PR when `action.yaml` or the test workflow changes, and on `workflow_dispatch`. There is no local test harness — the tests exercise real GitHub Actions runner behavior and the Checks API.

The test suite uses two verification patterns:
- **Inline assertions** — Steps within the matrix job that check local state (files in `/tmp/speedracer-*`, CPU model, check-name derivation).
- **Verify jobs** — Separate jobs (`verify-*`) that `needs:` the test job, wait for post-step propagation, then poll the Checks API to confirm check-run name, status, and conclusion.

Test scenarios cover: 2/3/4/8-copy matrices, slow/fast CPU combinations, failure paths, disabled mode, single-copy, multi-dimensional matrices, special characters in matrix values, ARM64 runners, TSV reading/fallback/override, and explicit vs auto-derived check-names.

To run tests: push to a branch that modifies `action.yaml` or the test workflow, or trigger manually via `gh workflow run speedracer-test.yaml`.

## State Files

The action communicates between the gate step and the post-step via files in `/tmp/`:
- `/tmp/speedracer-check-id` — The check-run ID (or empty if slow `a` that skipped posting)
- `/tmp/speedracer-check-name` — The resolved check name
- `/tmp/speedracer-copy` — The letter of this copy

## Development Notes

- Commit style: conventional commits (`fix:`, `refactor:`, `data:`, etc.)
- The `slow-cpu-pattern` resolution order is: caller input > `runner-pool.tsv` first data row > hardcoded fallback `"7763"`
- Copy `a` is special: it's the fallback that always runs (never CPU-gated), but it defers to any fast sibling that posts a check-run
- `kill -TERM $PPID` inside a composite action step produces `cancelled` status (not `failure`) — this is why the action uses composite steps rather than regular run steps
- The sibling polling loop has a short deadline (5s after an initial random 5-6s sleep) — it's a quick check, not a long poll
- `runner-pool.tsv` is committed by a bot only when the pool structure changes — don't manually edit it unless the profiler is broken
