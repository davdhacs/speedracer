# Speedracer

Speedracer is a GitHub Action for speculative execution of matrix jobs. Callers add `speedracer: [a, b]` (or more letters) as their last matrix dimension, then call this action as their first step. Multiple copies of the same job race; the fastest runner wins, duplicates cancel themselves.

The entire action is a single file: `action.yaml` (composite action with inline bash). There is no build step, no compiled language, no dependencies beyond `gh` and `jq`.

## How It Works

1. **Rank lookup** — Each copy reads `runner-pool.tsv` to find its CPU type's line number (rank). Line 1 = slowest = rank 1. Higher line = faster = higher rank. Unknown CPUs get rank 0 (worst).
2. **Check-run posting** — Every copy POSTs an `in_progress` check-run to the GitHub Checks API with rank embedded in the `external_id`: `spec-{letter}-r{rank}-{run_id}-{check_name}`.
3. **Sibling polling** — Each copy polls for siblings. Higher-ranked siblings always win. Same-rank ties break by letter (higher letter wins).
4. **Post-step** — Uses `gacts/run-and-post-run` to resolve the check-run after the job finishes. `job.status == success` → POST new completed/success check-run. `job.status == failure` → PATCH existing check-run to completed/failure.

The check-run contract is what makes this work with branch protection — the winning copy's check-run satisfies required status checks.

## Key Files

- `action.yaml` — The entire action: inputs, rank lookup, check-run management, post-step hook. All logic is inline bash.
- `runner-pool.tsv` — Auto-generated by the profiler workflow. Sorted slowest-first by CPU benchmark. The action reads this to assign ranks. Tab-separated: `cpu_id count pct avg_bench_ms avg_disk_mbs cores mem_mb full_model`. The profiler only commits this file when the CPU pool composition changes (new type appears or unseen slowest type is pruned) — benchmark number fluctuations are updated in-place but don't trigger commits.
- `.github/workflows/runner-profile.yaml` — Weekly cron (Monday 09:37 UTC) that benchmarks 10 runners and merges results into `runner-pool.tsv`. CPU types accumulate over time; unseen types are only pruned from the top (slowest) until a seen entry is reached.
- `.github/workflows/speedracer-test.yaml` — Comprehensive test suite that validates both job outcomes and Checks API state.
- `test-profiler.sh` — Local test suite for the profiler's filtering and merge logic (31 test cases).

## Testing

Tests run on push/PR when `action.yaml` or the test workflow changes, and on `workflow_dispatch`. There is no local test harness for the action — the tests exercise real GitHub Actions runner behavior and the Checks API.

The test suite uses two verification patterns:
- **Inline assertions** — Steps within the matrix job that check local state (files in `/tmp/speedracer-*`, rank, check-name derivation).
- **Verify jobs** — Separate jobs (`verify-*`) that `needs:` the test job, wait for post-step propagation, then poll the Checks API to confirm check-run name, status, and conclusion.

Test scenarios cover: 2/3/4/8-copy matrices, same-rank tiebreaks, failure paths, disabled mode, single-copy, multi-dimensional matrices, special characters in matrix values, ARM64 runners, TSV reading/missing/empty, rank from TSV vs rank 0, and explicit vs auto-derived check-names.

To run tests: push to a branch that modifies `action.yaml` or the test workflow, or trigger manually via `gh workflow run speedracer-test.yaml`.

For profiler tests: `bash test-profiler.sh` (runs locally, no GHA needed).

## State Files

The action communicates between the gate step and the post-step via files in `/tmp/`:
- `/tmp/speedracer-check-id` — The check-run ID
- `/tmp/speedracer-check-name` — The resolved check name
- `/tmp/speedracer-copy` — The letter of this copy
- `/tmp/speedracer-rank` — The CPU rank from the TSV (0 if unknown)

## Development Notes

- Commit style: conventional commits (`fix:`, `refactor:`, `data:`, etc.)
- Rank is determined by line position in `runner-pool.tsv` (1-indexed, data rows only). Higher rank = faster CPU. Rank 0 = unknown CPU, defers to everyone.
- No copy is special — all follow the same rank+letter priority. The highest-ranked copy with the highest letter wins.
- `kill -TERM $PPID` inside a composite action step produces `cancelled` status (not `failure`) — this is why the action uses composite steps rather than regular run steps
- The sibling polling loop has a short deadline (5s after an initial random 5-6s sleep) — it's a quick check, not a long poll
- `runner-pool.tsv` is committed by a bot only when the pool structure changes — don't manually edit it unless the profiler is broken
